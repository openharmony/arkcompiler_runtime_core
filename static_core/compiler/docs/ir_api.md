# IR APIs

Provide interface methods to build Panda IR and may be seen as three lowering modes
performing construction of graph parts during IR building, or an in-place rewrite
of an existing instruction.

These are based on the same Ruby DSL (.irt - IRTOC scripts), but produce slightly different
C++ output. API is selected via `--ir-api` (see irtoc options.rb).

The dispatching code is located in static_core/irtoc/lang/irtoc.rb.
A specified .irt file is parsed and requested C++ output is generated (see irtoc/lang/function.rb).

Currently, there are three supported IR APIs: *ir-constructor*, *ir-builder*, *ir-inline*

## ir-constructor

Its typical usage is as follows:
- standalone interpreter handlers
- fastpath/nativeplus entrypoints
- compiler unit tests

Functions `emit_ir` and `emit_body` in irtoc/lang/function.rb are responsible for processing
of .irt files and generation of C++ code. Resulting C++ code has the following structure:

```
    COMPILE(<name>) {
        // Some C++ code setting up a context: arch/mode/sources
        ...
        // C++ code constructing a graph
        GRAPH(GetGraph()) {
                ...
                INST(...);
                BASIC_BLOCK(..)
                ...
         }
    }
```

The code is placed into irtoc_code.cpp and/or irtoc_code_llvm.cpp (for the LLVM path).
The code relies on the macros (like GRAPH, INST, BASIC_BLOCK etc.) defined in
compiler/optimizer/ir/ir_constructor.h. During compilation the macros are expanded
into calls of IrConstructor methods to materialize a graph, and the codegen lowers
that graph to machine code.

`INST()` is as follows: `#define INST(ID, ...) builder_->NewInst(ID, __VA_ARGS__)`
Thus it is expanded into `IrConstructor::NewInst()`, which calls `Graph::CreateInst()`.

There are two forms of `Graph::CreateInst()`:
- **Form 1**: `CreateInst(opcode)` - creates `Inst` by opcode (a runtime parameter)
- **Form 2**: `CreateInst<opcode>(args…)` - creates `Inst` at compile time by template instantiation,
                where `opcode` is a template parameter and `args` are forwarded into a subclass constructor.

The main differences between these two forms are collected in the table below:

| Aspect            |          Form 1            |                 Form 2                   |
| ----------------- | -------------------------- | ---------------------------------------- |
| Opcode            | runtime parameter          | template parameter                       |
| Dispatch          | switch over OPCODE_LIST    | direct template instantiation            |
| Args              | opcode only                | forwarded to the subclass constructor    |
| Return type       | Inst* (base)               | BaseTypeOf<OPCODE>* (concrete subclass)  |
| Type/PC/imm setup | caller must set afterwards | provided via constructor args            |
| GC barrier flags  | must be set explicitly     | read, pre-/post-write set automatically  |
|                   |                            |                                          |

>`IrConstructor::NewInst()` calls **Form 1** of `Graph::CreateInst()`. In this case a user must
>explicitly set required properties of a newly created instruction. Special attention should be
>paid to GC barrier flags. In particular it is a user responsibility to explicitly set necessary
>barrier flags for non-managed Load/Store instructions and managed Load instructions. Managed Store
>instructions are processed automatically, see `IrConstructor::UpdateSpecialFlagsForReference()`
>for the details. A general recommendation is to set required barrier flags explicitly.
>
>Example 1 (from plugins/ets/irtoc_scripts/string.irt):
>```StoreI(tree_str, str1).Imm(Constants::TREE_STRING_LEFT_OFFSET).SetNeedWriteBarrier(false).ref```
>Example 2 (from plugins/ets/irtoc_scripts/string_builder.irt):
>```Store(buffer, slot_offs, char_array).SetNeedWriteBarrier(true).ref```

## ir-builder

Its typical usage is as follows:

- interpreter handlers defined in the corresponding irtoc plugin files (e.g. plugins/ets/irtoc_scripts/interpreter_handlers.irt)
- bytecode → IR translation (the InstBuilder pass) - bytecode ops are expanded into precomposed IR patterns
  by means of `InstBuilder::Build...()` family of methods (see compiler/optimizer/ir_builder/inst_builder.h)
- some simple intrinsics are generated by injecting corresponding IR into a graph (see compiler/optimizer/ir_builder/inst_builder-inl.h)

Function `generate_builder` in irtoc/lang/function.rb is responsible for processing of .irt files and generation of corresponding
irtoc_builder.cpp containing resulting C++ code.

Here is an example of generated C++ function:
```
    static Inst* BuildEcmaIstrue(Graph* graph, InstBuilder *inst_builder, BasicBlock** cur_bb, Inst* p_0, size_t pc, Marker marker) {
      auto* l_2 = graph->FindOrCreateConstant(DataType::Any(coretypes::TaggedValue::VALUE_TRUE));
      auto need_new_last_bb = (*cur_bb)->IsEmpty();
      auto* bb_2 = need_new_last_bb ? graph->CreateEmptyBlock() : (*cur_bb)->SplitBlockAfterInstruction((*cur_bb)->GetLastInst(), false);
      bb_2->SetGuestPc(pc);
      bb_2->CopyTryCatchProps(*cur_bb);
      if ((*cur_bb)->IsMarked(marker)) {
          bb_2->SetMarker(marker);
      }
      ...
      (*cur_bb) = bb_2;
      auto* l_1 = graph->CreateInstCompare(DataType::BOOL, pc);
      l_1->SetInput(0, p_0);
      l_1->SetInput(1, l_2);
      l_1->SetOperandsType(DataType::ANY);
      bb_2->AppendInst(l_1);
      auto* l_3 = graph->CreateInstCastValueToAnyType(DataType::ANY, pc);
      l_3->SetInput(0, l_1);
      l_3->SetAnyType(AnyBaseType::ECMASCRIPT_BOOLEAN_TYPE);
      bb_2->AppendInst(l_3);
      return l_3;
    }
```

>NOTE, `ir-builder` relies on **Form 2** of `Graph::CreateInst()`. In this case GC barrier flags are set
>*automatically* for all flavours of Load/Store instructions (see compiler/optimizer/ir/graph.h
>for the details).

## ir-inline

  It is to emit C++ functions like `bool IrtocInlineToBooleanDyn(Inst* source_inst)`, which rewrite
  source_inst (typically an intrinsic call) into a sequence of IR instructions (the source_inst is removed).

  Its typical usage is to generate interpreter handlers defined in the corresponding irtoc plugin files
  by means of `cpp_function`. Here is an example: `cpp_function(:"IrtocInlineToBooleanDyn")`

  The intrinsic-inlining pass (compiler/optimizer/optimizations/inline_intrinsics.cpp) iterates `Intrinsic`
  instructions and dispatches to the matching generated C++ function, which replaces the call in place.

  `GeneratorIrInline` (defined in irtoc/lang/ir_generator.rb) emits required C++ functions (usually these are placed
  into $BUILD_DIR/compiler/generated/irtoc_ir_inline.h.

  A typical shape of generated function is as follows:

```
      inline bool foo(Inst* source_inst) {
          Graph* graph = source_inst->GetBasicBlock()->GetGraph();
          auto pc = source_inst->GetPc();
          auto save_state = source_inst->GetSaveState();
          Inst* p_0 = source_inst->GetInput(0).GetInst();
          ...
          // create constants / insts / blocks etc.
          auto* l_3 = graph->FindOrCreateConstant(0);
          auto* l_1 = graph->CreateInstCastAnyTypeValue(DataType::INT32, pc);
          l_1->SetInput(0, p_0);
          auto* l_2 = graph->CreateInstCompare(DataType::BOOL, pc);
          l_2->SetInput(0, l_1);
          l_2->SetInput(1, l_3);
          l_2->SetOperandsType(DataType::INT32);
          ...
          source_inst->ReplaceUsers(<return value>);
          source_inst->GetBasicBlock()->RemoveInst(source_inst);
          return true;
      }
```
