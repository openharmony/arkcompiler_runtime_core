# Map / Set / Record Container Knowledge

This document only covers the internal implementation constraints of the Map, Set, and Record containers. For performance decisions, see `performance.md`; for the type system, see `type_system.md`.

## Core Model

Map is the underlying implementation. Set holds a `FinalMap<K,K>` delegation. Record directly inherits Map and constrains the key type.

## Map Internal Structure

Dual-array hash table: `data` (FixedArray, flattened key/val pairs) + `buckets` (ValueArray, embedded separate chaining linked list). Entries are appended in insertion order.

| Stage | Implementation |
| --- | --- |
| Hashing | `Runtime.getHashCodeByValue(k)` -> native intrinsic |
| Bucket index | `abs(hash) & (capacity - 1)`, where capacity is a power of 2 |
| Collision resolution | Separate chaining, front insertion |
| Key comparison | `sameValueZero()`: NaN === NaN, otherwise strict reference equality |

### Resizing (Grow / Shrink)

| Condition | Threshold | Action |
| --- | --- | --- |
| Grow | `numEntries * 4 >= cap * 3` (75%) | If tombstone >= 50%, compact only; otherwise double capacity |
| Shrink | `cap > 16 && sizeVal < cap / 4` (< 25%) | Halve capacity, no lower than `initialCapacity` |

### Deletion (Tombstone)

Buckets are marked with `DELETED_ENTRY`, and key/value are set to `undefined` (GC-friendly). `numEntries` is not decremented; `sizeVal` is decremented. Tombstones are compacted uniformly during the next rehash.

### Iterator Snapshot

When a rehash or clear occurs while there are active iterators, the old data is linked to the new data via `nestedData`, keeping the iterators valid.

## Set / Record Key Points

- Set: Zero independent hashing logic; all operations delegate to the internal `FinalMap`.
- Record: Directly inherits Map, with key type constrained to `Numeric | string | BaseEnum`. Supports bracket syntax via `$_get`/`$_set`. Does **not** have seal/freeze semantics.

## Constraints

- Map/Set/Record are all hand-written code, **not** autogenerated -- edit directly. Reason: no .j2 templates exist.
- Set **must not** introduce independent hashing logic. Reason: all hashing, resizing, and key comparison must remain consistent with Map behavior.
- Record key type constraints **must not** be relaxed at runtime. Reason: enforced by the compiler; runtime relaxation breaks type safety.
- Load factor thresholds (75%/50%/25%) **must not** be adjusted arbitrarily. Reason: these are performance-critical parameters; any change requires benchmark data.
- `sameValueZero` semantics **must not** be modified. Reason: must remain consistent with the ECMAScript specification.
- Map internal structure changes **must** be synchronized with `ets_map.h`. Reason: the C++ mirror layout must correspond one-to-one with the ETS layout.

## Pre-Modification Checklist

- Has `ets_map.h` been updated accordingly?
- Does the new key type have a corresponding hash dispatch in `GetHashCode()`?
- Do Set modifications preserve the delegation-to-Map pattern?
- Are iterator snapshot and tombstone handling correct?
